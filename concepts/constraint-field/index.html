When software hits the limit, we move the limit.

01 / The premise

Most people have learned to wait.

Wheels spin. Loading bars creep. We all agree that computers are slow.

But usually, the delay belongs to the software. Modern hardware can perform billions of operations a second, then sit idle behind a disk seek, network round trip, unnecessary copy or expensive abstraction. We measure the complete operation, find the mechanism imposing the delay, and remove it. An operation that took seconds can fall to milliseconds when the limiting mechanism disappears.

02 / Measured work

How we remove performance limits.

01 / rstat

System monitoring without the system-call churn.

rstat began as a Linux status-bar monitor that took too long to collect each update. We rebuilt its collection path around the work the kernel was already doing.

The optimized Rust poller still opened and parsed process files on every refresh. The current version uses an extended Berkeley Packet Filter (eBPF) program to account for tasks during scheduler events, then reads the accumulated counters in one userspace refresh.

02 / Archivist

222,242 entries, live in 420 milliseconds.

Archivist is a desktop frontend for curating, verifying and launching a large video-game archive. Its interface needed 222,242 filesystem entries ready before browsing began.

We designed one snapshot around its Rust producer and browser-based consumer. Parent indices replace repeated paths, while aligned columnar arrays replace per-entry objects. The Rust backend compresses that snapshot for transmission, and the browser reconstructs paths while viewing numeric columns in place.

03 / OpenZT2

Game data that's already in memory.

OpenZT2 is a clean-room, native reconstruction of Zoo Tycoon 2. Its engine needs an entire game's converted assets without copying them through a conventional runtime cache.

A launcher converts source assets into immutable, versioned records whose archived Rust layout can be read directly. The game maps that native content pack into its virtual address space, validates typed references into the mapped bytes, and asks Linux to load selected page groups within an available-memory budget.

04 / Rake In flight

A vector-first language for CPUs.

Rake is a research programming language for expressing single-instruction, multiple-data (SIMD) work directly. It makes a CPU vector, called a rack, the primitive unit of computation.

Rake's compiler type-checks rack operations without first reducing them to scalar code. It carries that vector structure into the Multi-Level Intermediate Representation (MLIR), giving later compiler stages enough information to target the CPU's vector instructions.
