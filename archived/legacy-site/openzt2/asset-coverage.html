Native Asset Coverage

The conversion boundary from original ZT2 source families to the small set of engine-native, memory-mappable asset families OpenZT2 actually wants.

Yes: one public core asset family per semantic thing is the right default. All original image formats should become one TextureAsset; BFB and NIF should become one ModelAsset; all clip codecs should be one AudioAsset; and so on. Tagged payload variants inside those types are still appropriate when delivery differs—for example resident PCM versus a streamable audio codec, or BC1 versus BC7 texture blocks.

Separate core types are justified where access and lifetime really differ: a model is not an animation clip, a material is not a shader, and chunk-streamed terrain is not a static model. Original extensions do not earn separate runtime types. Converter-only source parsers and discarded authoring sidecars never cross into the game.

The transport beneath these families is already accepted: one fixed-layout ContentProfile with a POD catalogue and page-aligned payload ranges, mapped read-only once. It is infrastructure rather than an asset family, so it is not included in the counts below.
