Binary Perimeter

The stripped Zoo Tycoon 2 binary's function, global/static, type identity, and byte-map recovery surfaces. Source: zt-function-perimeter.tsv, zt-global-perimeter.tsv, zt-type-coverage-reconciliation.tsv, and zt-byte-range-map.tsv. zt-vtable-methods-full.tsv supplies function-row overlay and containment validation.

Authority Model

The byte map is the address-fact archive. A function has an accepted exact identity only when a runtime-parent row in docs/analysis/zt-byte-range-map.tsv starts at that function entry and records recovered behavior. Region, provenance, vtable, and domain labels are triage context; they do not mean the function's concrete behavior has been recovered.

The type identity surface is built from the reconciliation table. It joins registered type-token cache handles, getter and parent/root evidence, named vtable rows, address-only vtable owner rows, foreign vtable getter uses, and zero-slot BFSafeHandle<T> rows. The global/static bar uses that table as a pivot so type-cache cells remain part of the static-address denominator while template-only type identities are counted explicitly. Identified non-type static cells are one bar segment; the family column explains their broad roles from accepted byte-map names.

Recovery Domains

Each bar below answers one completion question with one denominator. Solid bars use a closed denominator for that question. Striped bars use the currently known perimeter; future discoveries can expand the total. The byte-map bar is limited to the mapped runtime executable image at runtime-parent 0x00400000..0x00a92000. It reports bytes rather than row counts or partition counts.

Reading The Work Queue

Function recovery places accepted native function behavior, neutralized wrapper callsite/import rows, and still-unidentified function entries on the same frontier. The unidentified function entries are split by suspected structural domain from the call-graph/import propagation table. Vtable membership is an overlay on those same function entries; deploy validation requires every distinct vtable method body to appear in the function perimeter. That containment is the completion check, so the page does not render a separate vtable-method progress bar. Global/static and type identities combines referenced static cells with the template-only type rows needed to complete the known type surface.

Globals / Statics

The global/static table is generated from absolute references in .text. Type-cache and identified global names come from byte-map rows; unidentified cells remain raw referenced addresses until a byte-map fact names them. Template-only type identities are the zero-slot rows that have no corresponding static cache cell. The family column is explanatory grouping only; the name and status still come from the byte map.

Per-Class Method Overlay

Vtable method rows are generated from enumerated .rdata pointer runs and labelled from byte-map class evidence, early-slot type-getter evidence, constructor/vptr-store evidence, accepted byte-map function rows, and the function perimeter. Shared stubs and inherited helper bodies can occupy many slot rows while representing one function address. Each distinct method address belongs to the address-level function perimeter denominator.
