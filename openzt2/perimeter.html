Binary Perimeter

The stripped Zoo Tycoon 2 binary's function, global/static, type-cache, vtable, and byte-map recovery surfaces. Source: zt-function-perimeter.tsv, zt-global-perimeter.tsv, zt-vtable-methods-full.tsv, zt-rtti-classes.tsv, and zt-byte-range-map.tsv.

Authority Model

The byte map is the address-fact archive. A function has an exact identity record when a runtime-parent row in docs/analysis/zt-byte-range-map.tsv starts at that function entry. Native type getters are ordinary confirmed byte-map identities named get_type_*.

The function perimeter is a generated view over the stripped executable's function entries. It joins byte-map exact anchors with structural overlays: library provenance, vtable membership, and inferred domain. Detector proposals are produced live by tools/detect-address-id-candidates.py; durable review candidates and accepted identities live in the byte map.

Reading The Work Queue

Native function exact IDs track accepted byte-map names over the current analyzer function list. Type-cache bars distinguish named global cache cells from the separate question of whether cache rows, global rows, and getter callsites reconcile. Vtable bars track class/vtable naming and virtual slot identity separately. Byte-map partition coverage classifies non-overlapping map partitions as known-like or open/unknown-like, avoiding the misleading row-count denominator created by overlapping parent and child ranges.

Globals / Statics

The global/static table is generated from absolute references in .text. Type-cache and identified global names come from byte-map rows; unidentified cells remain raw referenced addresses until a byte-map fact names them.

Per-Class Method Perimeter

Vtable method rows are generated from enumerated .rdata pointer runs and labelled from byte-map class evidence, constructor/vptr-store evidence, and the function perimeter. They are a method-shape view, not a second accepted function-name archive.
