Binary Perimeter

The stripped Zoo Tycoon 2 binary's function, global/static, type-cache, vtable, and byte-map recovery surfaces. Source: zt-function-perimeter.tsv, zt-global-perimeter.tsv, zt-vtable-methods-full.tsv, zt-rtti-classes.tsv, and zt-byte-range-map.tsv.

Authority Model

The byte map is the address-fact archive. A function has an accepted exact identity only when a runtime-parent row in docs/analysis/zt-byte-range-map.tsv starts at that function entry. Candidate rows are proposed exact identities awaiting review. Region, provenance, vtable, and domain labels are triage context; they do not mean the function's concrete behavior has been recovered.

The class surface is the vtable surface unless stated otherwise. Vtable rows prove polymorphic native objects and their virtual slot counts. Additional name-only RTTI rows prove surviving type names, but they do not establish a vtable or virtual member perimeter. Type-cache rows track the runtime type-registration cache cells; parent relationships and getter linkage are separate facts over that same cache surface.

Reading The Work Queue

Native function behavior tracks accepted and candidate exact recovery over known native function entries. The current tables do not keep a suspected-only function-entry bucket. Class evidence separates vtable-backed classes from name-only RTTI rows, class names from vtable addresses, and total virtual slots from identified virtual slots. Type-cache handles count recovered cached type-handle cells once; global references, byte-map rows, and getter bodies are validation sources for that same fact. Type getter linkage and registered parent relationships are separate completion questions over those cache handles.

Globals / Statics

The global/static table is generated from absolute references in .text. Type-cache and identified global names come from byte-map rows; unidentified cells remain raw referenced addresses until a byte-map fact names them.

Per-Class Method Perimeter

Vtable method rows are generated from enumerated .rdata pointer runs and labelled from byte-map class evidence, constructor/vptr-store evidence, and the function perimeter. They are a method-shape view, not a second accepted function-name archive.
