Binary Perimeter

The stripped Zoo Tycoon 2 binary's function and global/static inventory, with per-function name, status, library provenance, vtable membership, and inferred domain. Source: zt-function-perimeter.tsv, zt-global-perimeter.tsv.

Authority Model

The byte map is the address-fact archive. A function has an exact identity record when a runtime-parent row in web/zt-byte-range-map.tsv starts at that function entry. Native type getters are ordinary confirmed byte-map identities named get_type_*.

The function perimeter is a generated view over the stripped executable's function entries. It joins byte-map exact anchors with structural overlays: library provenance, vtable membership, and inferred domain. Detector proposals are produced live by tools/detect-address-id-candidates.py; durable review candidates and accepted identities live in the byte map.

Reading The Work Queue

Closed/support rows are confirmed exact identities or generated/mapped support. Exact-ID candidates are recorded byte-map proposals awaiting review. Adapter-owned middleware rows carry third-party provenance such as Gamebryo, Lua, D3D9, zlib, libjpeg, or CRT. Product recovery frontier rows are remaining ZT/BlueFang functions that have vtable or domain context but still lack exact behavior names. Opaque rows have none of those facts.

Globals / Statics

The global/static table is generated from absolute references in .text. Type-cache and identified global names come from byte-map rows; unidentified cells remain raw referenced addresses until a byte-map fact names them.

Per-Class Method Perimeter

Vtable method rows are generated from enumerated .rdata pointer runs and labelled from byte-map class evidence, constructor/vptr-store evidence, and the function perimeter. They are a method-shape view, not a second accepted function-name archive.
