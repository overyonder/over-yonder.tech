Binary Perimeter

The stripped Zoo Tycoon 2 binary's function, global/static, type identity, vtable, and byte-map recovery surfaces. Source: zt-function-perimeter.tsv, zt-global-perimeter.tsv, zt-type-coverage-reconciliation.tsv, zt-vtable-methods-full.tsv, zt-rtti-classes.tsv, and zt-byte-range-map.tsv.

Authority Model

The byte map is the address-fact archive. A function has an accepted exact identity only when a runtime-parent row in docs/analysis/zt-byte-range-map.tsv starts at that function entry. Candidate rows are proposed exact identities awaiting review. Region, provenance, vtable, and domain labels are triage context; they do not mean the function's concrete behavior has been recovered.

The type identity surface is built from the reconciliation table. It joins registered type-token cache handles, getter and parent/root evidence, named vtable rows, address-only vtable owner candidates, foreign vtable getter uses, and zero-slot BFSafeHandle<T> rows into one denominator.

Recovery Domains

Each bar below answers one completion question with one denominator. Solid bars use a closed denominator for that question. Striped bars use the currently known perimeter; future discoveries can expand the total. The byte-map bar is limited to the mapped runtime executable image at runtime-parent 0x00400000..0x00a92000. It reports bytes rather than row counts or partition counts.

Reading The Work Queue

Native function behavior tracks accepted and candidate exact recovery over known original .text function entries in the deobfuscated executable. The current tables do not keep a suspected-only function-entry bucket. Function entries and class methods overlap: the vtable column records virtual-method membership when a function body appears in an enumerated vtable slot. Absence from that column does not prove the function is not a constructor, static member, non-virtual member, or helper associated with a type. Rows marked wrapper-indirected are depepper support records connected to the wrapper indirection bar. Global/static references is a separate address surface from native function entries. Type identity coverage reconciles the named type-cache handles from that global/static surface with vtable rows and zero-slot type rows.

Globals / Statics

The global/static table is generated from absolute references in .text. Type-cache and identified global names come from byte-map rows; unidentified cells remain raw referenced addresses until a byte-map fact names them.

Per-Class Method Perimeter

Vtable method rows are generated from enumerated .rdata pointer runs and labelled from byte-map class evidence, early-slot type-getter evidence, constructor/vptr-store evidence, accepted byte-map function rows, and the function perimeter. They are a method-shape view, not a second accepted function-name archive.
