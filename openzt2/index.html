OpenZT2 Reconstruction Dashboard

A map of the reconstruction pipeline: evidence, ownership/behavior, and implementation.

Pipeline

  1. Byte range map is the evidence gate by address/range: it records whether each mapped byte or range exists and what it is.
  2. Binary perimeter is the evidence pivot gate: it groups the recorded evidence by functions, type identities, vtables, globals, and broad evidence families.
  3. Functional specification is the ownership and behavior gate: it consumes recovered evidence and records which original systems own state/data and what behavior is understood or still missing.
  4. Reimplementation plan is the implementation gate: it consumes specified ownership/behavior and tracks what OpenZT2 has implemented, loaded, run, or still needs to build.

Pages

Byte Range Map

Contains the byte-level evidence ledger for the original executable: disk/runtime ranges, SafeDisc materialization records, accepted function identities, residual gaps, and status notes for exact address spans.

Binary Perimeter

Shows generated function, global/static, and type-identity inventories over the stripped binary, joining exact byte-map anchors with library provenance, vtable membership, inferred domains, and remaining frontier classifications.

Functional Specification

Records original-game ownership and behavior: source hydration, BF/BT framework surfaces, BFG substrate systems, ZT product domains, projection adapters, message registries, and remaining behavior gaps.

Reimplementation Plan

Tracks Rust/Bevy implementation progress for the specified ownership and behavior: what is planned, recorded, loaded, running, or adapted.
