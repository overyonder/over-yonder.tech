OpenZT2 Reimplementation Plan

Reconstruction Pipeline

  1. Byte range map is the evidence gate by address/range: it records whether each mapped byte or range exists and what it is.
  2. Binary perimeter is the evidence pivot gate: it groups the recorded evidence by functions, type identities, vtables, globals, and broad evidence families.
  3. Functional specification is the ownership and behavior gate: it consumes recovered evidence and records which original systems own state/data and what behavior is understood or still missing.
  4. Reimplementation plan is the implementation gate: it consumes specified ownership/behavior and tracks what OpenZT2 has implemented, loaded, run, or still needs to build.

This document is gate 4. It starts from the ownership and behavior described in ORIGINAL-GAME-FUNCTIONAL-SPECIFICATION.md and tracks the Rust/Bevy execution shape for that specified work.

The reconstruction keeps original source formats data-driven: Z2F archives, XML/MAXML/BFXML documents, localization, Lua, AI task/behavior files, materials, models, textures, audio, saves, scenarios, and packages. Rust may change storage mechanics and host integration. Original product concepts, manager ownership, message semantics, source schemas, and observable state transitions remain the authority.

The default implementation rule is to represent a specified concept directly in Bevy ECS/runtime types whenever Bevy can express the original behavior. Bevy States are available for finite, code-owned protocols only after the original behavior proves such a closed protocol exists. The current app flow does not use closed Bevy app/menu/game states: startup loading is the absence of RuntimeResourcesReady, and the original game shell owns the live flow through ECS-visible resources such as ModeRegistry, ActiveMode, ActiveLayoutPaths, ActiveKeymap, and CurrentCursor. Source-authored state such as shell-mode tokens, layout paths, hotkeys, cursors, and authored UI events stays data-driven: load it into registries/assets, then expose the live choice as ECS resources, components, entities, and messages. Domain ownership is exceptional: keep a concept in domain only when it is source-authored data, host-independent source or message behavior, pure rule, manager/native type identity, or another original surface that cannot live directly as Bevy components, resources, messages, or assets. Bevy runtime resources may store domain values as loaded records or semantic payloads; the owning live collection remains the Bevy resource, component, message stream, or asset.

Bevy State, Registries, and Source Modes

Data-driven state still needs a runtime fact somewhere. The target shape is one owner: source registries answer what the original data defines, ECS facts answer what is active right now, and systems perform the behavior.

ECS ownership does not mean every active effect becomes an entity/component. Bevy resources are ECS state too, and singleton-shaped resources are the right fit for genuinely global facts. The useful distinction is fact versus private runtime:

Res<ModeRegistry>   // source-authored definitions
Res<ActiveMode>     // current global fact: token plus resolved path
Res<ActiveLayoutPaths> // current global fact consumed by UI projection
Res<ActiveKeymap>   // current global fact derived from the active mode
Res<CurrentCursor>  // current global fact consumed by cursor presentation

Those resources are healthy when they store narrow facts with one owner and ordinary Bevy systems consume them. The unhealthy version is:

Res<ZTModeMgr>: ask me what mode we are in, what UI should show, what hotkeys
are active, what commands are queued, what effects should happen, and which
private transition rule wins.

That "ask me" shape is the shadow runtime. The problem is not that it is a resource. The problem is that it becomes a second scheduler/state machine beside Bevy.

For modes, the registry/catalogue holds source-authored definitions: mode tokens, parent paths, on/off events, layouts, cursors, hotkeys, and source payloads. The active mode is an ECS resource or component such as ActiveMode with the current token and resolved path. A transition system handles a ZT_SETMODE-style request by resolving the token against the registry, updating the ECS-owned active fact, and publishing concrete effects such as ModeEntered, ModeExited, LayoutRequested, CursorChanged, and KeymapActivated. UI, input, cursor, audio, and gameplay systems react to those facts and messages through ordinary Bevy schedules and run conditions.

A registry is healthy when it stores source definitions. An ECS resource or component is healthy when it stores the current live fact. A shadow runtime appears when one retained manager owns the current mode, active layouts, active hotkeys, command queues, transition side effects, and private dispatch rules, then other systems ask that manager what app they are in. That pattern recreates a second runtime beside ECS. It should be collapsed into registries, ECS facts, messages, and systems.

The design history matters because the code has already moved through these shapes:

  • f6966524 (Start game reconstruction workspace, 2026-06-14) created the pre-Bevy domain-first scaffold. It had source/domain/host boundaries, host-independent UI contract records, and only a tiny placeholder GameMode::{MainMenu, LoadingWorld, InWorld}. It was an architectural starting point before recovered data-driven runtime behavior existed.
  • a3c0142f (Set up Bevy reconstruction shell, 2026-06-14) introduced the Bevy host shell.
  • b6317954 (Replace fake app phase with domain UI dispatch, 2026-06-14) moved behavior into ZTAppRuntime and UiRuntimeContext, where UI messages such as ZT_SPLASH_CLOSED changed an active layout. That was source-aware, and it also put live app/UI state in a domain-side shadow runtime.
  • f0d35051 (Sync shell mode transitions to native owners, 2026-06-16) recovered the more useful semantic shape: source-token mode transitions resolved through a shell-mode catalogue, computed mode paths, on/off events, layouts, cursors, and hotkeys. The semantic direction was right; the live owner still needs to be ECS rather than a domain manager.
  • By 6c95b926 (Split Cargo and flake roots by project, 2026-06-19), the Bevy runtime had closed menu/in-game mode variants into States/SubStates and mapped source messages into those Rust enums. That improved native Bevy ownership for finite lifecycle phases, and it overreached when it turned source-authored modes into fixed Rust state variants.

The intended final arrangement is therefore: use closed Bevy States only when the recovered original contract is genuinely finite and code-owned; load source-authored modes and UI layouts as data; keep the current source-selected mode/layout as ECS-visible facts; express behavior as systems reacting to messages, events, components, resources, and assets.

The current Bevy runtime follows that shape. The top-level scheduler keeps only phase-oriented ordering sets such as InputCollection, RuntimeMessageDispatch, Simulation, PresentationSync, and RenderPreparation. Runtime systems carry explicit run conditions: startup/loading work runs before RuntimeResourcesReady, normal runtime work uses runtime_resources_ready, and loaded-world systems add concrete ECS gates such as in_loaded_world. The former closed OpenZt2AppSet menu/game/overlay buckets have been removed.

The current refactor boundary is: no private game_runtime module, no RuntimeSupport resource, seed type, or compatibility namespace, no root feature compatibility prelude, and no compatibility owner that other systems ask for unrelated facts. FeatureRuntimePlugin only composes feature-owned registrations. Startup uses explicit seed/composition values such as StartupResourceSeeds and startup_sources::HostSources while source documents are being hydrated; once runtime resources are published, live facts sit in narrower resources such as runtime configuration, session/profile/time, diagnostics, external/web state, random state, mode registry, active mode, active layouts, active keymap, and current cursor. Their code owners are concrete core modules for configuration, session, profile, time, diagnostics, external/web, and startup source hydration. UI source events are recorded in order as UiSourceEventRecorded, then decoded by runtime systems into the typed effects owned by their feature boundary. This is not the final completion of every UI message family; it is the ownership rule for finishing those families. The shared core message module owns only the ordered source-event clock/path and the runtime-ready marker; feature effects such as mode/input, world, time, terrain, undo, path, zoopedia, audio, camera, photo, AI, animal, staff, save, web, and renderer requests live beside their owning runtime modules.

The work board below uses actual owners. Product concepts, runtime boundaries, source-format families, and host boundaries may all own rows when they carry real implementation work. Aggregate progress buckets are invalid owners.

Active Cleanup Goal

The active cleanup goal is the native Bevy runtime cleanup checkpoint. Its durable work-board owner is Bevy runtime / Refactor checkpoint / Feature boundary cleanup; adjacent rows inherit work only when the change crosses their real owner boundary.

Completion means closing the remaining refactor-cleanup class in code, or moving the residue to the owning row as explicit unrecovered contract work. The checkpoint is complete when:

  • documented commands and aliases used as acceptance evidence resolve in the current workspace before they count as a checkpoint gate;

Expansion Rule

Add or split a row when the functional specification gives a distinct implementation owner: a manager, native class family, source file family with a consumer, state transition, message handler group, formula, persisted surface, or adapter edge. Unspecified ownership, behavior, payloads, and formulae stay in the functional specification. Work-board rows track implementation of specified ownership/behavior and link back to specification gaps when implementation is blocked.
