OpenZT2 Reimplementation Plan

ORIGINAL-GAME-FUNCTIONAL-SPECIFICATION.md owns the original Zoo Tycoon 2 contract: source-owned data, native classes, managers, messages, state transitions, and adapter boundaries. This plan owns the current Rust execution shape for that contract.

The reconstruction keeps original source formats data-driven: Z2F archives, XML/MAXML/BFXML documents, localization, Lua, AI task/behavior files, materials, models, textures, audio, saves, scenarios, and packages. Rust may change storage mechanics and host integration. Original product concepts, manager ownership, message semantics, source schemas, and observable state transitions remain the authority.

Ownership Boundaries

Boundary Current Rust owner Job
Source formats and source classification domain::source_format, domain::source_asset, domain::source_support Parse original files, classify source documents, model BFResourceMgr, BFXML*, BFXMLSchemaMgr, BFLocaleMgr, BFTypedBinder, BFRegistry, BFScriptContextMgr, BFUserProfileMgr, and BFTimeKeeper contracts.
Product semantics domain::* modules named for product concepts Define source-derived values, pure rules, manager contracts, message tokens, and open native surfaces. Domain modules do not own live runtime state, retained runtime collections, queues, caches, app/session state, or presentation state.
Live runtime and projection bevy crate Own Bevy-native ECS components/resources/entities/messages, schedules, current app/session state, runtime collections, asset handles, Kira/mlua/renderer/window adapters, input routing, and presentation.

The default implementation rule is to represent a recovered concept directly in Bevy ECS/runtime types whenever Bevy can express the original concept without losing the recovered contract. Domain ownership is exceptional: keep a concept in domain only when it is source-authored data, host-independent source or message contract, pure rule, manager/native type identity, or another original surface that cannot live directly as Bevy components, resources, states, messages, or assets. Bevy runtime resources may store domain values as loaded contracts or semantic payloads; the owning live collection remains the Bevy resource, component, state, message stream, or asset.

The work board below uses actual owners. Product concepts, runtime boundaries, source-format families, and host boundaries may all own rows when they carry real implementation work. Aggregate progress buckets are invalid owners.

Expansion Rule

Add or split a row when the specification, source data, or Rust code gives a distinct implementation owner: a manager, native class family, source file family with a consumer, state transition, message handler group, formula, persisted surface, or adapter boundary. Attach unresolved behavior, payloads, and formulae to that owning row. If the owner is unclear, keep the fact in the functional specification until source, code, or accepted contract evidence gives it a blue-team implementation home.
