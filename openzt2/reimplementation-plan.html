OpenZT2 Reimplementation Plan

ORIGINAL-GAME-FUNCTIONAL-SPECIFICATION.md describes the original game domain: the source-owned files, native managers, object types, message contracts, state transitions, and behaviours the reconstruction must preserve. This document describes the engineering shape of the Rust runtime: how those contracts should be stored, loaded, indexed, simulated, and presented efficiently in a modern Bevy engine.

The specification answers "what did Zoo Tycoon 2 mean and require?" This file answers "how do we execute those same meanings without building a slow facade over a 2008 single-threaded C++ heap model?"

The reconstruction must remain backwards-compatible with the original source formats: Z2F archives, XML/MAXML/BFXML documents, localization files, materials, models, textures, audio, saves, and packages. It does not need to preserve the original C++ memory layout, allocator behaviour, STL container choices, pointer graph, vtable mechanics, or duplicate transient trees unless one of those mechanics is part of an observable or persisted contract.

Expansion Rule

Binary perimeter rows and source assets are recovery inputs. Add a work-board row when a distinct manager, source file family, state transition, formula, message handler group, or adapter boundary appears. Attach unresolved methods, globals, payloads, and formulae to the nearest owning row. If ownership is unclear, leave the behavior blank until the source/spec evidence gives it a home.

Supporting Adapter Reference

Bevy and library surfaces are support machinery for the plan above. They are not workbook domains and should not become top-level checklist buckets.
Bevy feature mapping

This mapping records which host/engine features are expected to carry original-game contracts. It is an adapter checklist, not a second implementation taxonomy.

Adapter surfaceUse in OpenZT2Boundary rule
Crate boundariescontent owns source bytes and parsed source documents; domain owns host-independent semantic values and pure rules; bevy owns live runtime state, ECS entities/resources, fixed handlers, rendering, audio, input, and asset handles.Original ZT*, BF*, and BFG* names describe preserved semantics, not a requirement to preserve C++ storage ownership.
Runtime substrateBevy app, ECS, schedules, states, time, tasks, events/messages, queries, commands, observers, and change detection provide the canonical mutable runtime.Recovered native handlers mutate Bevy resources/components and may call pure domain helpers; they should preserve original tokens, sender, target, payload, and ordering.
Asset and source loadingZ2F-backed asset sources/readers and extension loaders map zt2:// paths into Bevy assets such as Image, model/scene assets, materials, animation clips, audio sources, Lua scripts, and structured documents.Do not extract every asset or mirror Bevy asset storage into domain caches. Store source ids, path ids, asset ids, or handles.
Documents and localizationXML/BFXML/MAXML and locale sources should be parsed into compact ordered representations that preserve attributes, mixed content, ids, and lookup/fallback behavior.Hydrators consume source documents; runtime state lives in the owning runtime domain.
UI, input, and modesBevy UI/text/picking/input host original UI layouts, widgets, visual state, interaction, focus, cursor, and mode tools.Clicking a widget dispatches authored original events and recovered handlers. It should not execute invented Bevy-only behavior.
World, terrain, and selectionWorld instances become Bevy entities/components; terrain uses packed arrays, chunks, dirty regions, generated meshes, and picking for brush/placement hits.Terrain should not be one entity per tile, and the runtime should not maintain a duplicate world graph beside ECS.
Renderer, materials, animation, and cameraBevy image, mesh, material, shader, PBR, animation, scene, camera, lighting, post-processing, gizmo, and diagnostic surfaces replace Gamebryo/D3D mechanics behind preserved component inputs.Source-visible render contracts still come from the functional specification: BFSceneGraphComponent, BFAnim*, BFSpriteMgr, material, texture, and camera command semantics.
Audio and scriptingKira/Bevy audio execute recovered BFSND_* commands and sound-tag policy; mlua executes shipped Lua behind recovered BFS_* bindings.Backend playback and Lua host mechanics are replaceable; command arguments, returns, errors, side effects, categories, and timing are product contracts.
Host, platform, and toolingWindowing, cursor, IME, filesystem, registry/profile paths, networking/download/MOTD, logging, diagnostics, CI, file watching, inspectors, and debug gizmos are host adapters.Use these when they support fidelity, recovery, or development. They do not define semantic game domains.

Storage Rules

  • Immutable authored data is stored once and shared by id or handle.
  • Mutable runtime state is stored once in Bevy resources/components.
  • Presentation is derived from runtime state and updated through dirty sets, generation counters, changed components, or explicit messages.
  • Duplicate source trees, loaded-document stores, asset registries, entity graphs, and presentation caches are removed unless they are a proven original persisted/runtime contract.

Connection Pattern

Source asset -> Bevy loader/resource; input event -> original message handler; runtime state change -> presentation sync; asset handle ready -> attach image, mesh, material, audio, animation, or script; simulation tick -> mutate Bevy components/resources.
